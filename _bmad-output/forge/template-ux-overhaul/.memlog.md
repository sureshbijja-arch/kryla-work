- (assumption) Custom made rich UI = each Kryla page should feel individually designed, not like a template system
- (lock) Direction: system elevates average content — even a member with 3 photos and 2 services looks like they hired an agency. Design does the heavy lifting, not the member's content quality.
- (lock) Kryla needs a proper design foundation first: type scale, spacing system, color token relationships — before more section variants or templates.
- (lock) Retrofit now — rewrite all section components on the foundation before production data exists. No migration debt.
- (lock) Foundation implementation: CSS custom properties (readable by designers, works outside Tailwind) + Tailwind config extension (enforces system in classnames). Both, not either.
- (lock) One design system, persona-switched ratios. C (warm craft) as base. A (editorial) for photographers/doctors. Shared token names across all — persona changes the ratio and spacing density, not the token structure.
- (lock) New design_mode column on pages table: craft | editorial | product. Set at onboarding from persona. No temporary bridge — proper column from the start. Palette stays separate (color only). design_mode owns spatial grammar.
- (lock) auto variant resolved at render time in LayoutRenderer — reads member content, picks variant. Not written back to DB.
- (lock) auto variant content signals: (1) has gallery photo → photo variant, (2) avatar only, no gallery → split or centered, (3) no photos at all → dark typographic. System picks, admin can override.
- (lock) Build sequence: W1 foundation (CSS tokens + design_mode column, invisible), W2 Hero rewrite + auto variant (first visible win), W3 remaining sections on tokens, W4 persona default section arrays. Do not skip W1.
